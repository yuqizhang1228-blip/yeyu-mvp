# 夜屿：Braintrust 提示词评测流程（Playbook）

> 目的：把提示词迭代从“凭感觉”变成**可自动回归、可对比、可定位样本**的工程流程。  
> Project：`yeyu_test`

---

## 0. 总览：我们在做什么

我们用本地脚本调用 **DeepSeek（与你线上一致的调用方式）**生成对话输出，然后把：
- 每个测试用例的输入/输出
- 每个指标（硬指标 + 主观评审）
- 每轮对话的 span（turn_1/turn_2/…）

写入 Braintrust，形成 Experiments，可在网页端对比版本与回归。

Braintrust 的作用是：**记录、可视化、对比、筛选最差样本、持续评估**。  
LLM 实际生成对话：**当前由你自己的 DeepSeek 来跑**（更贴近线上真实体验）。

---

## 1. 两套测试集：Smoke vs Regression

### 1.1 Smoke（冒烟测试集）

- **目标**：快、底线、别炸  
- **使用场景**：每次你改提示词一两句，都先跑 smoke，看有没有明显回归  
- **覆盖重点**：
  - 禁用语是否出现
  - 是否连问多个问题（审讯感）
  - `<card>` 是否还能解析、结构是否正确
  - 安全边界是否能触发
  - 问身份/短回复/跑偏等“容易翻车”的入口
- **数据集文件**：`braintrust/datasets/yeyu_smoke.jsonl`

### 1.2 Regression（主回归集）

- **目标**：稳、贴近真实高频、能支撑“这版真的更好”  
- **使用场景**：每次大版本（v3.2/v3.3…）或关键改动后跑一次  
- **覆盖重点**（按夜屿真实高频）：
  - **职业困境（合并）**：职场迷茫/事业发展困惑（选择困境、价值感崩塌、能力焦虑、路径不确定等）
  - **生活重大变故**：裁员/家人重病/亲人离世/负债/事故等
  - **亲密关系破碎**：分手/背叛/冷暴力/离婚恐惧/被抛下感等
  - 必要边界用例：问身份、短回复、跑偏、安全边界
- **要求**：包含足量 **10+ 轮长对话**用例，才能稳定评估“推进质量”和“收束出卡片”
- **数据集文件**：`braintrust/datasets/yeyu_regression.jsonl`（第一版骨架，后续持续扩充）

## 2. 一键跑测试：命令与开关

### 2.1 环境变量（本地）

放在项目根目录 `.env.local`（已被 `.gitignore` 忽略）：

- `BRAINTRUST_API_KEY="..."`
- `DEEPSEEK_API_KEY="..."`
- `BRAINTRUST_PROJECT="yeyu_test"`

可选开关：
- `YEYU_SUITE="smoke" | "regression"`（默认 smoke）
- `YEYU_LIMIT="10"`（只跑前 N 条，用于快速迭代）
- `BRAINTRUST_EXPERIMENT="自定义实验名"`（可读性更好）

### 2.2 跑 smoke

```bash
npm run eval:yeyu
```

或显式：

```bash
YEYU_SUITE=smoke npm run eval:yeyu
```

### 2.3 跑 regression

```bash
YEYU_SUITE=regression npm run eval:yeyu
```

## 3. 评估维度（Metrics）：我们到底在测什么

### 3.1 硬指标（Deterministic / 更客观）

这些指标由脚本规则计算（不是 Braintrust“自己判断”）：

1) **禁用语（命中=0）**  
- 含义：是否命中“绝对禁止的话”列表（鸡汤/空话/模板口癖等）  
- 期望：长期接近 100%

2) **单问题约束（≤1问号）**  
- 含义：一条回复里是否抛了多个问题（多问会造成审讯感）  
- 期望：> 95%（越高越好）

3) **卡片JSON可解析**  
- 含义：若应出卡片，`<card>...</card>` 内 JSON 是否能 `JSON.parse()`  
- 期望：100%

4) **卡片结构完整（thought/reframe/actions）**  
- 含义：是否具备 thought/reframe/actions，且 actions 恰好两条、非空  
- 期望：100%

5) **卡片仅出现一次**  
- 含义：`<card>` 标签是否只出现一次（避免解析/展示异常）  
- 期望：100%

6) **行动具体性（时间+对象+动作）**  
- 含义：actions 是否可执行、够具体（规则打分 0~1）  
- 期望：> 0.80

> 备注：当 `expected.card_required=true` 时，没出卡片会直接记 0 分，从而确保卡片相关指标能在 Braintrust Summary 中出现并可对比。

### 3.2 主观评审（LLM Judge / 半客观）

我们用一个“裁判模型”评审输出，但为了降低漂移，裁判采取：
- **只评两项**：人感、推进质量
- **四档选择题**：A/B/C/D（不自由打分）
- **必须引用证据**：reasons 必须引用输出中的具体片段
- **雷区硬扣分**：鸡汤/连问/强模板出现则最高只能给 C

映射得分：
- A = 1
- B = 0.75
- C = 0.4
- D = 0

指标名：**主观评审（LLM Judge）**

## 4. 如何解读 Braintrust 实验结果（最实用的读法）

### 4.1 不要只看总分，先看“回归点”

每次跑完实验，优先看：
- 哪些指标下降了（例如“单问题约束”从 95% 掉到 80%）
- 哪些样本在下降指标里最差（score=0 的那几条）
- 这些失败样本是否集中在某个 `scene/subscene`

### 4.2 典型故障→对应改法

- **单问题约束下降**：提示词需要更硬的“每条只问一个问题”约束 + 单问句式模板  
- **卡片解析/结构不稳定**：提示词需要更强“收束触发条件 + 输出格式硬约束”  
- **行动具体性偏低**：提示词需要更强“动作必须含时间+对象+动作”示例与禁用泛泛话

### 4.3 核心闭环（持续优化）

1) 选 1 个最差指标（例如单问题约束）  
2) 在 Braintrust 找 3 条最差样本（带 turn spans）  
3) 针对失败形态改提示词（小改）  
4) 跑 Smoke 验证没炸  
5) 跑 Regression 验证整体趋势变好  

---

## 5. 为什么我们做 turn spans（专业模式）

每个 case 会生成一个根 span：`case:<id>`  
下面是每轮模型调用的子 spans：`turn_1/turn_2/...`

这样你在 Braintrust 点开一条 case，能看到：
- 每轮用户输入
- 每轮模型输出
- 每轮耗时与错误

用途：当指标回归时，能定位“在哪一轮开始崩”，而不是只看最后一句。

---

## 6. 维护规范（建议）

- 提示词每次改动都给一个版本号（v3.2/v3.3…）
- Regression 集随线上反馈持续扩充（尤其是“职业困境/重大变故/关系破碎”的子类）
- 每周至少跑一次 Regression，形成稳定趋势
